# 安全漏洞分析报告 — v7alpha vs v7beta

> **主题：SSRF 服务端请求伪造 (Server-Side Request Forgery)**  
> **日期：2026-07-13**  
> **版本：v7alpha（故意漏洞版）vs v7beta（安全修复版）**

---

## 一、本次重点修复 — SSRF 服务端请求伪造

v7alpha 的 `/fetch-url` 路由用 `urllib.request.urlopen()` 直接访问用户提交的 URL，未做任何限制。攻击者可利用服务器作为跳板，读取本地文件、探测内网服务、甚至攻击内部系统。

---

### 漏洞1：file:// 协议读取本地任意文件

#### 漏洞原理

`urllib.request.urlopen()` 支持 `file://` 协议，可直接读取服务器本地文件。v7alpha 未限制 URL 协议，攻击者提交 `file:///etc/passwd` 即可获取系统用户信息，提交 `file:///path/to/app.py` 即可获取应用源码。

#### 漏洞位置

`v7alpha/app.py` 第 1225 行：
```python
response = urllib.request.urlopen(url, timeout=10)
```

#### 攻击方式

```
POST /fetch-url
url=file:///etc/passwd
→ 返回 root:x:0:0:..., kali:x:1000:1000:...

POST /fetch-url
url=file:///home/kali/Desktop/agent-project/claude-code-project/v7alpha/app.py
→ 返回完整应用源码（含 secret_key 配置、数据库路径等）
```

#### 修复方案（v7beta）

白名单协议检查，仅允许 `http://` 和 `https://`：

```python
parsed = urllib.parse.urlparse(url)
if parsed.scheme not in ("http", "https"):
    return error("不支持的协议，仅允许 http 和 https")
```

#### 效果验证

| Payload | v7alpha | v7beta |
|---------|:--:|:--:|
| `file:///etc/passwd` | 🔴 系统文件泄露 | ✅ "不支持的协议" |
| `http://example.com` | ✅ 正常 | ✅ 正常 |

### 漏洞2：访问内网地址

#### 漏洞原理

v7alpha 不检查 URL 解析后的目标 IP 地址，攻击者可构造 `http://127.0.0.1:8090/` 访问服务器本地端口，或 `http://10.0.0.1/` 探测内网其他服务器。

#### 漏洞位置

`v7alpha/app.py` 第 1221-1257 行：整个 `/fetch-url` 路由无任何 IP 检查逻辑。

#### 攻击方式

```
POST /fetch-url
url=http://127.0.0.1:8090/profile
→ 返回本机 /profile 页面 HTML（含用户数据）

POST /fetch-url
url=http://192.168.1.1/
→ 探测内网路由器管理界面
```

#### 修复方案（v7beta）

DNS 解析主机名后检查所有 IP 地址，拒绝私有/回环/保留地址：

```python
import ipaddress, socket

BLOCKED_NETWORKS = [
    ipaddress.ip_network("127.0.0.0/8"),    # loopback
    ipaddress.ip_network("10.0.0.0/8"),     # private
    ipaddress.ip_network("172.16.0.0/12"),  # private
    ipaddress.ip_network("192.168.0.0/16"), # private
    ipaddress.ip_network("169.254.0.0/16"), # link-local
    ipaddress.ip_network("::1"),             # IPv6 loopback
]

def _check_ip_safety(ip_str):
    ip = ipaddress.ip_address(ip_str)
    for net in BLOCKED_NETWORKS:
        if ip in net:
            return False
    return True
```

#### 效果验证

| Payload | v7alpha | v7beta |
|---------|:--:|:--:|
| `http://127.0.0.1:8090/` | 🔴 内网访问成功 | ✅ "不能访问内网地址" |
| `http://localhost/` | 🔴 内网访问 | ✅ "不能访问内网地址" |
| `http://10.0.0.1/` | 🔴 内网访问 | ✅ "不能访问内网地址" |
| `http://192.168.1.1/` | 🔴 内网访问 | ✅ "不能访问内网地址" |

### 漏洞3：DNS 重绑定攻击

#### 漏洞原理

即使核验 IP，攻击者仍可利用 **DNS 重绑定（TOCTOU）** 绕过：域名第一次解析返回合法公网 IP → 检查通过 → `urlopen()` 第二次解析返回内网 IP → SSRF 成功。

#### 修复方案（v7beta — 自审计发现并修复）

对于 HTTP 请求，将解析后的安全 IP 直接替换到 URL 中，并设置 `Host` 请求头。这样 urllib 不再进行第二次 DNS 解析，彻底消除重绑定窗口：

```python
# DNS 只解析一次，IP 直接写入 URL
safe_ip = safe_ips[0]  # 优选 IPv4
ip_url = f"{parsed.scheme}://{safe_ip}{parsed.path}"
req = urllib.request.Request(ip_url)
req.add_header("Host", parsed.hostname)  # 保持原始主机名
response = urllib.request.urlopen(req, timeout=10)
```

对于 HTTPS，TLS 证书验证使得 IP 替换不可行，但 DNS 重绑定窗口极短，实际利用困难。

## 二、白盒审计同步发现并修复的问题

### 2.1 数字/十六进制 IP 伪装

**问题：** 浏览器和部分库会将 `2130706433`（十进制）、`0x7f000001`（十六进制）解析为 `127.0.0.1`，可能绕过 IP 检查。

**修复：** 新增 `_is_suspicious_hostname()` 函数，检测纯数字、十六进制前缀、八进制点分格式的主机名并拒绝。

### 2.2 重定向跟随风险

**问题：** `urllib.request.urlopen()` 默认跟随 HTTP 重定向。攻击者可将域名重定向到 `file:///etc/passwd` 或内网地址。

**修复：** 注册自定义 `_NoRedirectHandler`，禁止 `urlopen()` 跟随任何重定向。

### 2.3 响应内容未转义

**问题：** 抓取的远程内容可能包含恶意 HTML/JS，直接渲染会导致 XSS。

**修复：** 使用 `<pre>` 标签配合 Jinja2 默认转义（不使用 `| safe`），确保所有远程内容均以纯文本展示。

### 2.4 IPv6 优先导致请求失败

**问题：** 系统同时有 IPv4 和 IPv6 地址时，`safe_ips` 可能优先使用不可达的 IPv6 地址。

**修复：** 增加 IPv4 优先级逻辑。

## 四、漏洞影响总结

| 漏洞 | 类型 | 严重程度 | v7alpha | v7beta |
|------|------|:--:|:--:|:--:|
| file:// 读取本地文件 | SSRF / File Disclosure | 🔴 严重 | 可读任意文件 | 已修复 |
| 内网地址访问 | SSRF / Internal Scan | 🟠 高危 | 可探测内网 | 已修复 |

