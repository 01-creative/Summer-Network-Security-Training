# 安全漏洞分析与修复报告

---

## 一、本次重点修复 — SSRF 服务端请求伪造

**SSRF（Server-Side Request Forgery，服务端请求伪造）** 是一种利用服务端发起请求的漏洞，攻击者通过控制服务端URL参数，诱使服务器向内部网络或本地文件系统发起未授权的请求，从而获取敏感信息、攻击内部服务或执行端口扫描。

在 v7alpha 版本中，新增的 URL 抓取功能 `fetch_url` 存在三类典型的 SSRF 漏洞。

---

### 漏洞1：file:// 协议读取本地文件

| 项目 | 内容 |
|------|------|
| **严重程度** | 高危 |
| **CVSS 3.1** | 7.5 (AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:N/A:N) |
| **位置** | `v7alpha/app.py` 第 1241 行 |
| **类型** | SSRF -> 任意文件读取 |

#### 漏洞原理

Python 的 `urllib.request.urlopen()` 默认支持 `file://` 协议，可读取服务器上的任意本地文件。攻击者只需在前端的 URL 输入框中输入 `file:///etc/passwd`，服务端就会不加校验地执行该请求，将文件内容返回给攻击者。

**漏洞代码（v7alpha/app.py，第 1239-1244 行）：**

```python
try:
    # 【漏洞】直接使用用户提供的 URL，不做任何校验
    resp = urllib.request.urlopen(url, timeout=10)
    status_code = resp.getcode()
    content = resp.read(5000).decode("utf-8", errors="replace")
    fetch_result = f"状态码: {status_code}\n\n内容:\n{content}"
```

#### 攻击复现

**攻击 payload：**
```
url=file:///etc/passwd
```

**攻击效果（实际测试验证）：**

```
状态码: None

内容:
root:x:0:0:root:/root:/usr/bin/zsh
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
bin:x:2:2:bin:/bin:/usr/sbin/nologin
...
kali:x:1000:1000::/home/kali:/usr/bin/zsh
```

该攻击成功泄露了完整的系统 `/etc/passwd` 文件，包含所有系统用户和普通用户的账户信息。

**攻击 payload 2：读取应用源代码**
```
url=file:///home/kali/Desktop/agent-project/claude-code-project/v7alpha/app.py
```

攻击效果：成功读取完整的 `app.py` 源代码，泄露业务逻辑、数据库结构、密码哈希算法、密钥等敏感信息。

#### 修复方案

**禁止非 http/https 协议（建议代码，第 1239 行之前添加）：**

```python
from urllib.parse import urlparse

# URL 协议白名单校验
parsed = urlparse(url)
if parsed.scheme not in ("http", "https"):
    return "仅支持 http 和 https 协议", 400
```

#### 效果验证

| 测试 | 修复前 | 修复后 |
|------|--------|--------|
| `file:///etc/passwd` | 返回文件内容 | 返回"仅支持 http 和 https 协议" |
| `file:///app.py` | 返回源代码 | 返回"仅支持 http 和 https 协议" |

### 漏洞2：访问内网地址

| 项目 | 内容 |
|------|------|
| **严重程度** | 高危 |
| **CVSS 3.1** | 8.6 (AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:L/A:L) |
| **位置** | `v7alpha/app.py` 第 1241 行 |
| **类型** | SSRF -> 内网探测/攻击 |

#### 漏洞原理

`urllib.request.urlopen()` 发起请求前不检查目标 IP 地址是否为私有/回环地址。攻击者可利用此漏洞扫描内网、访问管理后台、攻击内部服务（如 Redis、MySQL、Kubernetes API Server 等未授权端点）。

**漏洞代码（v7alpha/app.py，第 1241 行）：**

```python
resp = urllib.request.urlopen(url, timeout=10)
```

没有任何 IP 地址过滤逻辑。`127.0.0.1`、`10.x.x.x`、`172.16-31.x.x`、`192.168.x.x` 均可直接访问。

#### 攻击复现

**攻击 payload 1：访问内部管理端点**
```
url=http://127.0.0.1:8090/profile
```

**攻击效果（实际测试验证）：**

```
状态码: 200

内容:
[profile页面的HTML内容……]
```

成功访问了本地 8090 端口上的 `/profile` 端点，该端点需要登录才能访问，但服务端发起请求时使用的是本地回环地址，绕过了外部访问限制。

**攻击 payload 2：探测内网服务**
```
url=http://127.0.0.1:8090/admin
url=http://10.0.0.1:8080/
url=http://192.168.1.1:80/
```

攻击者可根据响应时间、状态码、响应内容判断端口是否开放及服务类型。

#### 修复方案

**对目标 IP 进行 DNS 解析和私有地址检查（第 1241 行之前添加）：**

```python
import socket
from urllib.parse import urlparse

def is_private_ip(ip):
    """检查 IP 是否为私有/回环/保留地址"""
    parts = ip.split(".")
    if len(parts) != 4:
        return True
    # 127.0.0.0/8 回环
    if parts[0] == "127":
        return True
    # 10.0.0.0/8 私有A类
    if parts[0] == "10":
        return True
    # 172.16.0.0/12 私有B类
    if parts[0] == "172" and 16 <= int(parts[1]) <= 31:
        return True
    # 192.168.0.0/16 私有C类
    if parts[0] == "192" and parts[1] == "168":
        return True
    # 169.254.0.0/16 链路本地
    if parts[0] == "169" and parts[1] == "254":
        return True
    # 0.0.0.0/8
    if parts[0] == "0":
        return True
    return False

parsed = urlparse(url)
hostname = parsed.hostname
try:
    ip = socket.gethostbyname(hostname)
    if is_private_ip(ip):
        return "不支持访问内网地址", 400
except socket.gaierror:
    return "无法解析域名", 400
```

#### 效果验证

| 测试 | 修复前 | 修复后 |
|------|--------|--------|
| `http://127.0.0.1:8090/profile` | 返回内部页面内容 | 返回"不支持访问内网地址" |
| `http://10.0.0.1/` | 尝试连接内网服务 | 立即拒绝返回错误 |

### 漏洞3：无协议限制

| 项目 | 内容 |
|------|------|
| **严重程度** | 高危 |
| **CVSS 3.1** | 9.1 (AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:N) |
| **位置** | `v7alpha/app.py` 第 1241 行 |
| **类型** | SSRF -> 任意协议利用 |

#### 漏洞原理

Python 的 `urllib.request.urlopen()` 除了支持 `file://` 外，还支持 `ftp://`、`gopher://`、`dict://` 等协议。其中 `gopher://` 协议尤其危险——攻击者可通过 gopher 协议构造任意 TCP 数据包，向 Redis、MySQL、Memcached 等 TCP 服务注入恶意命令。

**漏洞代码（v7alpha/app.py，第 1241 行）：**

```python
resp = urllib.request.urlopen(url, timeout=10)
```

没有任何协议白名单限制。

#### 攻击复现

**潜在攻击 payload 1：gopher 攻击内部 Redis（如 6379 未授权）**
```
url=gopher://127.0.0.1:6379/_*2%0d%0a$4%0d%0a...
```

**潜在攻击 payload 2：FTP 协议探测**
```
url=ftp://anonymous@127.0.0.1:21/
```

**潜在攻击 payload 3：dict 协议探测端口**
```
url=dict://127.0.0.1:3306/INFO
```

#### 修复方案

**白名单仅允许 http/https（与漏洞1修复一致）：**

```python
from urllib.parse import urlparse

ALLOWED_SCHEMES = {"http", "https"}

parsed = urlparse(url)
if parsed.scheme not in ALLOWED_SCHEMES:
    return "仅支持 http 和 https 协议", 400
```

#### 效果验证

| 测试 | 修复前 | 修复后 |
|------|--------|--------|
| `file:///etc/passwd` | 读取任意文件 | 被白名单拦截 |
| `ftp://127.0.0.1/` | 尝试 FTP 连接 | 被白名单拦截 |
| `gopher://127.0.0.1:6379/` | 可构造 TCP 攻击 | 被白名单拦截 |
| `dict://127.0.0.1:3306/` | 可探测 MySQL 端口 | 被白名单拦截 |
| `http://example.com` | 正常访问 | 正常访问（白名单通过） |

---

### SSRF 防御最佳实践总结

防御 SSRF 应采取"纵深防御+最小权限"策略，多个层面叠加防护：

| 防御层级 | 具体措施 |
|----------|----------|
| **协议白名单** | 仅允许 `http://` 和 `https://`，拒绝所有其他协议 |
| **DNS解析+IP检查** | 解析域名后检查是否为私有/回环/链路本地/保留地址，拒绝内网IP |
| **域名黑名单** | 拒绝访问 127.0.0.1、localhost、[::1]、内部域名等 |
| **HTTP重定向检查** | 跟随重定向时重新验证目标 URL，防止绕过 |
| **最小网络权限** | 使用最小权限容器/网络策略，禁止服务端出站到内网 |
| **超时限制** | 设置合理的请求超时（已实现 timeout=10） |
| **响应大小限制** | 限制响应读取大小（已实现 read(5000) 限制） |

## 二、其他同步发现并修复的问题

### 2.1 v7alpha 与 v6beta 代码对比

从整体架构看，v7alpha 在 v6beta 基础上仅增加了 SSRF 漏洞相关的 URL 抓取功能代码，其他安全机制保持完全一致。

**文件变更清单：**

| 文件 | 变更内容 |
|------|----------|
| `app.py` | 新增第5-6行（urllib 导入）、第1216-1257行（fetch_url 路由） |
| `templates/index.html` | 新增第83-106行（URL抓取UI） |

### 2.2 v6beta 已有安全机制回顾

v6beta 版本已实施的安全措施在 v7alpha 中完整保留，以下逐一评估：

#### 2.2.1 SQL注入防护（状态：已修复）
- **问题**：所有 SQL 查询均使用参数化查询（`?` 占位符）
- **位置**：`app.py` 第 56-57 行 `PRAGMA`、第 77-89 行 `CREATE TABLE`、第 102-113 行 `INSERT`、第 168-170 行 `SELECT`、第 208-210 行 `INSERT` 等全部数据库操作
- **评价**：参数化查询完全消除 SQL 注入风险

#### 2.2.2 密码安全（状态：已修复）
- **密码存储**：使用 `werkzeug.security.generate_password_hash`（PBKDF2+SHA256，自动加盐）
- **强度校验**：第 599-611 行 `validate_password_strength()` 要求 8位+大小写+数字+特殊字符
- **时序侧信道防御**：第 457 行引入假哈希值，不存在用户使用假哈希比对
- **密钥长度校验**：第 341-345 行校验 FLASK_SECRET_KEY >= 32 字符

#### 2.2.3 CSRF 防护（状态：已修复）
- 所有 POST 表单均携带 CSRF Token
- 第 489-512 行 `generate_csrf_token()` + `validate_csrf()` 使用 `secrets.compare_digest()` 进行安全比对
- 登录、注册、改密、退出、充值、上传、URL抓取（新增）全部受 CSRF 保护

#### 2.2.4 文件上传安全（状态：已修复）
- 扩展名白名单（第 369 行）
- MIME 类型验证（第 1107-1110 行）
- 魔数验证（第 379-407 行 `validate_image_content()`）
- `secure_filename()` 防路径穿越（第 1113 行）
- UUID 前缀防覆盖枚举（第 1139 行）
- 文件大小显式检查（第 1127-1130 行）

#### 2.2.5 路径穿越防护（状态：已修复）
- `/page` 路由使用 `os.path.realpath()` 严格校验路径（第 1187-1191 行）
- 确保访问路径在 `PAGES_DIR` 范围内

#### 2.2.6 会话管理（状态：已修复）
- `HttpOnly=True` 防 XSS 窃取
- `SameSite=Lax` 防 CSRF
- `session_version` 机制确保改密后旧会话失效（第 660-669 行）
- 首次登录强制改密（第 649-656 行）

#### 2.2.7 登录限流（状态：已修复）
- 双窗口限流（IP 维度 + 用户名维度）
- IP 5次/5分钟锁定
- 同一用户 10次/15分钟锁定
- 短窗口 60秒内3次失败锁定60秒

#### 2.2.8 响应头安全（状态：已修复）
- `Server` 头移除
- `X-Content-Type-Options: nosniff`
- `X-Frame-Options: DENY`
- `X-XSS-Protection: 1; mode=block`
- `Strict-Transport-Security`（HTTPS 时启用）

### 2.3 本次新增功能的改进建议

v7alpha 新增的 URL 抓取功能在以下方面需要改进：

1. **SSRF 缺陷**：仍需增加协议白名单、内网 IP 检查（详见第一部分）
2. **前端显示改进**：结果区域可增加加载状态提示、清除按钮
3. **用户权限限制**：可考虑仅允许管理员使用 URL 抓取功能
4. **请求频率限制**：建议对 `/fetch-url` 端点增加单独的请求频率限制
