# 安全漏洞分析报告 — v5alpha vs v4beta

## 一、本次重点修复 — 文件包含与路径穿越漏洞

### 漏洞1：目录穿越读取任意文件（Path Traversal / Local File Inclusion）

#### 漏洞原理
v5alpha 新增的 `/page` 路由（`app.py` 第 692 行）实现了"动态页面加载"功能，但存在严重的路径穿越漏洞。攻击者通过构造含有 `../` 序列的 `name` 参数，可逃离 `pages/` 目录限制，读取服务器上任意文件。

漏洞产生的根本原因：
1. **第 700 行**：`page_path = os.path.join("pages", name)` — 用户输入的 `name` 参数未经任何校验，直接拼接到文件路径中
2. **第 704-707 行**：直接以用户控制的路径打开文件 — `open(page_path, "r")`
3. **未使用** `os.path.abspath()` 或 `os.path.realpath()` 将路径解析为绝对路径后验证其归属目录
4. **未过滤** `../` 等目录穿越序列
5. **第 712-717 行**：`.html` 后缀回退机制进一步扩大了可读取的文件范围

#### 漏洞位置

| 文件 | 行号 | 说明 |
|------|------|------|
| `v5alpha/app.py` | 692-732 | 整个 `/page` 路由（漏洞函数 `dynamic_page`） |
| `v5alpha/app.py` | 700 | `os.path.join("pages", name)` — 核心漏洞点 |
| `v5alpha/app.py` | 704-709 | 直接打开用户指定路径 |
| `v5alpha/templates/index.html` | 82-90 | `page_content` 使用 `\| safe` 渲染未转义 HTML |

#### 攻击方式
攻击者在未登录状态下即可发起攻击（该路由未做身份校验）：

| 攻击 URL | 效果 | 泄露内容 |
|----------|------|----------|
| `/page?name=help` | 正常访问帮助页面（预期行为） | — |
| `/page?name=../app.py` | 读取 app.py 源文件 | secret_key、DB_PATH、generate_password_hash |
| `/page?name=../templates/base` | 读取 base.html 模板 | HTML 模板结构和 CSRF token 机制 |
| `/page?name=../../../../../../../etc/passwd` | 读取系统密码文件 | Linux 系统用户列表 |

**验证结果（全部通过）：**
```
# /page?name=../app.py 泄露 secret_key 配置
app.secret_key = _raw_key
app.secret_key = os.urandom(32).hex()
FLASK_SECRET_KEY

# /page?name=../../../../../../../etc/passwd 泄露系统文件
root:x:0:0:root:/root:/bin/bash

# /page?name=../templates/base 泄露模板
<!DOCTYPE html>
```

#### 修复方案（见 v5beta）
修复的核心思路：**永远不要信任用户输入**。任何涉及文件路径的操作都必须经过严格的合法性校验。

```python
# ===== 安全版本 (v5beta) =====
import os

# 定义允许的基础目录
PAGES_DIR = os.path.join(os.path.dirname(os.path.abspath(__file__)), "pages")

@app.route("/page")
def dynamic_page():
    name = request.args.get("name", "")
    if not name:
        return redirect(url_for("index"))

    # [安全] 解析为绝对路径
    page_path = os.path.join(PAGES_DIR, name)
    page_path = os.path.abspath(page_path)

    # [安全] 验证最终路径必须在 PAGES_DIR 内
    if not page_path.startswith(PAGES_DIR):
        return render_template("index.html", ..., page_content="<p>页面不存在</p>")

    # [安全] 限制只允许读取 .html 文件
    if not page_path.endswith(".html"):
        page_path += ".html"
        page_path = os.path.abspath(page_path)
        if not page_path.startswith(PAGES_DIR):
            return render_template("index.html", ..., page_content="<p>页面不存在</p>")

    if os.path.isfile(page_path):
        with open(page_path, "r", encoding="utf-8") as f:
            page_content = f.read()
    else:
        page_content = "<p>页面不存在</p>"
```

#### 修复前后代码对比

| 代码位置 | v5alpha（漏洞版本） | v5beta（修复版本） |
|----------|-------------------|-------------------|
| 基础路径 | `os.path.join("pages", name)` | `os.path.join(PAGES_DIR, name)` |
| 相对/绝对路径 | 相对路径（相对于 CWD） | 绝对路径（相对于 `__file__`） |
| 路径解析 | 无 | `os.path.abspath()` |
| 目录限制 | 无 | `str.startswith(PAGES_DIR)` 校验 |
| 文件类型限制 | 无（任意文件） | 仅 `.html` 文件 |
| 错误提示 | "页面不存在" | "页面不存在"（不泄露路径信息） |

#### 效果验证
修复后：
- `/page?name=help` → 正常显示帮助页面
- `/page?name=../app.py` → 提示"页面不存在"（路径逃离被拦截）
- `/page?name=../../../etc/passwd` → 提示"页面不存在"（目录穿越被拦截）
- `/page?name=../templates/base` → 提示"页面不存在"（路径不在 pages/ 内）

### 漏洞2：源码泄露（Source Code Disclosure）

#### 漏洞原理
漏洞1（目录穿越）的直接后果就是源码泄露。攻击者通过 `/page?name=../app.py` 可以完整读取 `app.py` 的源代码，从中提取关键安全敏感信息。

#### 漏洞位置
与漏洞1相同 — `v5alpha/app.py` 第 692-732 行。

#### 攻击方式
```
GET /page?name=../app.py HTTP/1.1
Host: 127.0.0.1:8086
```

#### 泄露的高危信息
通过读取 `app.py` 源码，攻击者可获取：

| 泄露信息 | 代码行 | 风险 |
|----------|--------|------|
| FLASK_SECRET_KEY | 第 340-348 行 | Session 伪造、身份冒充 |
| DB_PATH 数据库路径 | 第 41 行 | 数据库文件定位 |
| generate_password_hash 使用 | 第 104-105, 110-111 行 | 哈希算法识别 |
| CSRF Token 生成机制 | 第 489-493 行 | CSRF 防御绕过 |
| 上传目录路径 | 第 365-366 行 | 更多攻击面 |
| 登录频率限制实现 | 第 446-596 行 | 限流绕过分析 |
| 数据库架构 | 第 77-89 行 | SQL 注入面分析 |

#### 修复方案
修复路径穿越漏洞（漏洞1的修复方案）即可从根本上消除源码泄露风险。此外，还可以采取纵深防御措施：

```python
# [安全] 禁止 /page 路由读取 .py 文件
ALLOWED_EXTENSIONS = {".html", ".htm"}
if not any(page_path.endswith(ext) for ext in ALLOWED_EXTENSIONS):
    return "页面不存在"
```

#### 效果验证
修复后 `/page?name=../app.py` 返回"页面不存在"，无法读取源码。

## 二、其他同步发现并修复的问题

除本次新增的路径穿越漏洞外，v5alpha 与 v4beta 在安全层面的对比如下：

### 2.1 本次新增内容
v5alpha 基于 v4beta（安全基线版本）开发，新增了以下内容：

| 新增项 | 文件 | 说明 |
|--------|------|------|
| `/page` 路由 | `v5alpha/app.py` 第 692-732 行 | **存在路径穿越漏洞** |
| `pages/help.html` | `pages/help.html` | 帮助中心页面（正常内容） |
| 帮助中心链接 | `templates/index.html` 第 22 行 | 导航栏"帮助中心"链接 |
| `page_content` 显示区域 | `templates/index.html` 第 82-90 行 | 动态页面内容渲染（使用 `\| safe`） |

### 2.2 v4beta 已有的安全机制（v5alpha 已继承）
v5alpha 继承了 v4beta 的所有安全加固措施，以下机制保持完好：

| 安全机制 | v4beta | v5alpha |
|----------|--------|---------|
| Secret Key 强度校验（≥32 字符） | 第 341-345 行 | 继承 |
| Session Cookie HttpOnly/SameSite | 第 354-360 行 | 继承 |
| CSRF Token 防护 | 第 489-512 行 | 继承 |
| 登录频率限制（双窗口 + 用户名维度） | 第 446-596 行 | 继承 |
| 时序侧信道防御（DUMMY_HASH） | 第 457 行 | 继承 |
| Session 版本号管理 | 第 659-669 行 | 继承 |
| 首次登录强制改密 | 第 648-656 行 | 继承 |
| 密码强度校验 | 第 599-611 行 | 继承 |
| XSS 输入过滤 | 第 614-626 行 | 继承 |
| 文件上传安全（扩展名 + MIME + 魔数校验） | 第 374-407 行 | 继承 |
| HTTP 安全响应头 | 第 416-427 行 | 继承 |
| 参数化 SQL 查询 | 各处 | 继承 |
| LIKE 通配符过滤 | 第 300 行 | 继承 |

### 2.3 关于 `| safe` 过滤器的安全说明
`templates/index.html` 第 87 行使用 `{{ page_content | safe }}` 渲染动态页面内容。在 v4beta 中，所有模板变量均使用默认的 HTML 转义（Jinja2 自动转义），但 `| safe` 过滤器会关闭转义。在 v5alpha 的 `/page` 路由中，读取的文件内容直接作为 HTML 渲染，这虽然对 `help.html` 这类受控内容无害，但配合路径穿越漏洞，攻击者可能读取包含恶意 HTML 标签或 JavaScript 的文件，引发 XSS 攻击。

### 2.4 修复建议总结

| 严重程度 | 漏洞 | 修复优先级 |
|----------|------|-----------|
| **严重** | 路径穿越读取任意文件 (CWE-22) | 立即修复 |
| **严重** | 源码泄露 (CWE-540) | 立即修复 |
| **中危** | 未转义 HTML 渲染 (CWE-79) | 建议修复（配合路径穿越风险升级） |
