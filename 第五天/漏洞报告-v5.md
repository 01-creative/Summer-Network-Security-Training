# 安全漏洞分析报告 — v5alpha vs v5beta

> **主题：文件包含漏洞 (Path Traversal / Local File Inclusion)**  
> **日期：2026-07-13**  
> **版本：v5alpha（故意漏洞版）vs v5beta（安全修复版）**

---

## 一、本次重点修复 — 文件包含与路径穿越漏洞

v5alpha 新增的 `/page` 路由实现了"动态页面加载"功能。但由于用户输入 `name` 参数未经任何校验直接拼接到文件路径，引发了**目录穿越（Path Traversal）**漏洞，攻击者可读取服务器上任意文件。

---

### 漏洞1：目录穿越读取任意文件（Path Traversal / Local File Inclusion）

#### 漏洞原理

v5alpha 的 `/page` 路由（`app.py` 第 692-732 行）存在三个核心问题：

1. **直接拼接用户输入到路径**：`os.path.join("pages", name)` — `name` 直接来自 `request.args`
2. **无路径校验**：不调用 `os.path.abspath()`，不过滤 `../`，不验证目录边界
3. **使用相对路径基准**：`"pages"` 是相对于 CWD 的相对路径，运行目录不同可能导致不同的攻击面

```
用户输入: ?name=../../../etc/passwd
拼接结果: pages/../../../etc/passwd
实际路径: /etc/passwd  ← 逃离了 pages/ 目录！
```

#### 漏洞位置

| 文件 | 行号 | 说明 |
|------|------|------|
| `v5alpha/app.py` | 692-732 | 整个 `/page` 路由 |
| `v5alpha/app.py` | 700 | `os.path.join("pages", name)` — 核心漏洞点 |
| `v5alpha/app.py` | 704-709 | `open(page_path)` 直接打开用户控制路径 |

#### 攻击方式

**验证通过的全部攻击向量：**

| 攻击 URL | 结果 | 泄露内容 |
|----------|------|----------|
| `/page?name=help` | 正常显示 | 帮助页面（预期行为） |
| `/page?name=../app.py` | **源码泄露** | `secret_key`、`DB_PATH`、所有路由逻辑 |
| `/page?name=../templates/base` | **模板泄露** | HTML 结构、CSRF token 机制 |
| `/page?name=../../../etc/passwd` | **系统文件泄露** | Linux 系统用户列表 |

通过读取 `app.py` 源码，攻击者可获取：FLASK_SECRET_KEY 配置方式、数据库路径、哈希算法、所有路由和安全机制的实现细节。

#### 修复方案（v5beta）

v5beta 采用**三步纵深防御**修复路径穿越：

**第一步 — 解析绝对路径：**
```python
PAGES_DIR = os.path.join(os.path.dirname(os.path.abspath(__file__)), "pages")
candidate = os.path.realpath(os.path.join(PAGES_DIR, name))
```
使用项目根目录的 `pages/` 绝对路径作为基准（而非相对路径 `"pages"`），`os.path.realpath()` 同时解析符号链接。

**第二步 — 目录边界校验（含前缀碰撞防护）：**
```python
if not candidate.startswith(PAGES_DIR + os.sep) and candidate != PAGES_DIR:
    return "页面不存在"
```
使用 `PAGES_DIR + os.sep`（即 `.../pages/`）而非 `PAGES_DIR`（即 `.../pages`），防止 `pages-backup` 等同名前缀目录绕过。

**第三步 — 异常捕获：**
```python
try:
    ...  # 路径操作
except (ValueError, OSError, UnicodeError):
    return "页面不存在"
```
捕获空字节注入（`\x00`）、权限错误、编码攻击引起的异常，统一返回安全提示。

#### 修复代码对比

| 关键点 | v5alpha（有漏洞） | v5beta（已修复） |
|--------|-------------------|------------------|
| 基准路径 | `"pages"`（相对路径） | `PAGES_DIR`（从 `__file__` 计算的绝对路径） |
| 路径解析 | 无 | `os.path.realpath()` |
| 目录边界 | 无校验 | `startswith(PAGES_DIR + os.sep)` |
| 异常处理 | 无 | `try/except (ValueError, OSError, UnicodeError)` |
| 错误信息 | "页面不存在" | "页面不存在"（不泄露路径） |
| 符号链接 | 跟随（未检查） | 解析后校验（超出 pages/ 即拒绝） |

#### 效果验证

| 测试用例 | v5alpha | v5beta |
|----------|---------|--------|
| `/page?name=help` | ✅ 正常显示 | ✅ 正常显示 |
| `/page?name=help.html` | ✅ 加 `.html` 后显示 | ✅ 加 `.html` 后显示 |
| `/page?name=../app.py` | 🔴 **泄露源码！** | ✅ "页面不存在" |
| `/page?name=../../../etc/passwd` | 🔴 **泄露系统文件！** | ✅ "页面不存在" |
| `/page?name=../templates/base` | 🔴 **泄露模板！** | ✅ "页面不存在" |
| `/page?name=nonexistent` | ✅ "页面不存在" | ✅ "页面不存在" |

### 漏洞2：源码泄露（Source Code Disclosure）

#### 漏洞原理

漏洞1（目录穿越）的直接后果就是**源码泄露**。攻击者通过 `/page?name=../app.py` 可完整读取 `app.py` 的约 1200 行源代码，从中提取：

| 泄露信息 | 风险 |
|----------|------|
| `app.secret_key` 来源 | 了解密钥生成机制，辅助 Session 伪造 |
| `DB_PATH` 数据库路径 | 定位数据库文件实施进一步攻击 |
| `generate_password_hash` 调用 | 识别哈希算法（scrypt/pbkdf2） |
| CSRF Token 生成函数 | 了解 CSRF 防御实现 |
| 登录限流参数 | 分析绕过限流的策略 |
| 上传文件白名单 | 了解允许的文件类型 |
| 数据库表结构 | 了解用户表字段 |

#### 攻击方式

```
GET /page?name=../app.py HTTP/1.1
Host: 127.0.0.1:8086
→ 返回 app.py 完整源码（~1200 行）
```

#### 修复方案

与漏洞 1 相同 — 路径边界校验从根本上阻止了 `../app.py` 的读取。另外 v5beta 还可以增加文件后缀限制作为纵深防御：
```python
if not candidate.endswith(".html"):
    return "页面不存在"
```

#### 效果验证

修复后 `/page?name=../app.py` → "页面不存在"，源码泄露路径被封堵。

## 二、白盒审计同步发现并修复的问题

v5beta 在自审计阶段额外发现并修复了以下问题：

### 2.1 前缀碰撞攻击（Prefix Collision）

**问题：** 原始实现使用 `candidate.startswith(PAGES_DIR)` 校验目录边界。如果存在 `pages-backup/` 等同名前缀目录，攻击者可能绕过校验。

**修复：** 改为 `candidate.startswith(PAGES_DIR + os.sep)`，强制要求路径分隔符紧随目录名：
```python
# 修复前（可被绕过）
if not candidate.startswith(PAGES_DIR):
    
# 修复后（安全）
if not candidate.startswith(PAGES_DIR + os.sep) and candidate != PAGES_DIR:
```

### 2.2 空字节异常处理

**问题：** Wapiti 扫描器注入 `\x00` 空字节到 `name` 参数，`os.path.realpath()` 在 Python 3 中对此抛出 `ValueError`，导致未捕获异常返回 HTTP 500。

**修复：** 整个路径操作包裹在 `try/except (ValueError, OSError, UnicodeError)` 中：
```python
try:
    candidate = os.path.realpath(os.path.join(PAGES_DIR, name))
except (ValueError, OSError, UnicodeError):
    return render_template("index.html", ..., page_content="<p>页面不存在</p>")
```

### 2.3 多编码路径穿越测试（全部通过）

v5beta 自审计对所有已知编码绕过向量进行了测试：

| 攻击向量 | Payload | 结果 |
|----------|---------|:--:|
| 标准 `../` | `../app.py` | ❌ 被拦截 |
| 多点穿越 | `....//` | ❌ 被拦截（`....` 是合法目录名） |
| URL 编码 | `%2e%2e%2fapp.py` | ❌ Flask 解码后被拦截 |
| 双重编码 | `%252e%252e%252f` | ❌ Flask 不解二次编码 |
| Unicode | `..∕` | ❌ Linux 不识别 Unicode 分隔符 |
| 符号链接到 `/etc/passwd` | `evil_link` | ❌ `os.path.realpath()` 解析后拒绝 |

### 2.4 v4beta 已有安全机制保持完好

v5alpha 和 v5beta 均继承自 v4beta 的安全基线，以下机制验证通过：

| 安全机制 | 状态 |
|----------|:--:|
| CSRF 防护（所有 POST） | ✅ |
| 密码哈希（scrypt） | ✅ |
| 登录频率限制（IP+用户名双维度） | ✅ |
| 时序侧信道防御（DUMMY_HASH） | ✅ |
| Session 版本号（改密后旧 Session 失效） | ✅ |
| 首次登录强制改密 | ✅ |
| 参数化 SQL（无注入） | ✅ |
| XSS 输入过滤（sanitize_input） | ✅ |
| 文件上传安全（扩展名+MIME+魔数） | ✅ |
| HTTP 安全响应头 | ✅ |
| LIKE 通配符过滤 | ✅ |

### 2.5 关于 `| safe` 过滤器的安全说明

`templates/index.html` 使用 `{{ page_content | safe }}` 渲染动态页面内容。`| safe` 关闭 Jinja2 的自动 HTML 转义，这本身有 XSS 风险。但 v5beta 中 `page_content` 仅来源于 `pages/` 目录内受信任的 `.html` 文件（路径校验保证了这一点），因此实际风险可控。生产环境中建议改为使用 `{{ page_content }}` 并让 HTML 标签经转义显示，或对动态内容做白名单标签过滤。
