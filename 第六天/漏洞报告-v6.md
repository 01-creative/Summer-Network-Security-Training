# 安全漏洞分析报告 — v6alpha vs v6beta

> **主题：密码修改安全漏洞（认证绕过 + CSRF + 水平越权）**  
> **日期：2026-07-13**  
> **版本：v6alpha（故意漏洞版）vs v6beta（安全修复版）**

---

## 一、本次重点修复 — 密码修改安全漏洞

v6alpha 的 `/change-password` 路由存在三个连环漏洞：**任意用户密码篡改**、**无需验证原密码**、**CSRF 保护缺失**。攻击者只需普通用户身份即可窃取管理员账户。

---

### 漏洞1：任意用户密码篡改（无身份校验 / 水平越权）

#### 漏洞原理

v6alpha 的 `/change-password` 从 POST 表单的 `username` 字段直接读取目标用户名，**不验证该用户名是否与当前登录会话匹配**。任意已登录用户可通过修改隐藏字段或构造 HTTP 请求，篡改系统中任意用户的密码。

```python
# v6alpha：username 从表单获取，无身份匹配校验
username = request.form.get("username", "")
# 直接用表单中的 username 更新密码
conn.execute("UPDATE users SET password = ? WHERE username = ?", (hash, username))
```

#### 漏洞位置

| 文件 | 行号 | 说明 |
|------|------|------|
| `v6alpha/app.py` | 890 | `username = request.form.get("username")` — 身份从客户端提交 |
| `v6alpha/app.py` | 902 | `WHERE username = ?` — 不验证身份匹配 |
| `v6alpha/templates/profile.html` | 58 | `<input type="hidden" name="username" value="{{ user.username }}">` — 隐藏字段可篡改 |

#### 攻击方式

1. alice 登录系统，访问个人中心页面
2. 用浏览器开发者工具将隐藏字段 `username` 改为 `admin`
3. 输入新密码 `hacked123!`，点击提交
4. admin 的密码被改为 `hacked123!`，alice 用此密码登录获取管理员权限

#### 修复方案（v6beta）

身份标识仅从 `session["uid"]` 获取，完全忽略表单中的任何 username 参数：

```python
# v6beta：身份仅从 session 获取
uid = session.get("uid")
if not uid:
    return redirect(url_for("login"))
# 修改当前用户的密码，不接受表单中的 username
db_update_user_password(uid, generate_password_hash(new_password))
```

#### 修复代码对比

| 关键点 | v6alpha（有漏洞） | v6beta（已修复） |
|--------|-------------------|------------------|
| 目标用户来源 | `request.form.get("username")` | `session["uid"]` |
| 身份验证 | 无（信任客户端提交） | 绑定到登录会话 |
| 表单隐藏字段 | `username` 隐藏域 | 无此字段 |

#### 效果验证

```
v6alpha: alice POST username=admin&new_password=hacked → admin 密码被改 ✅（漏洞）
v6beta:  alice POST username=admin&new_password=hacked → 参数被忽略，仅改自己的密码 ✅（安全）
```

### 漏洞2：无需验证原密码即可改密

#### 漏洞原理

v6alpha 的改密流程不要求输入当前旧密码。攻击者只要知道了目标用户名（通过注册页面枚举即可获取），并在已登录状态下提交，就能覆盖该用户的密码。这意味着：

- 攻击者不需要知道目标用户的原始密码
- 结合漏洞1，任何已登录用户都可以替换任意人的密码

#### 漏洞位置

`v6alpha/app.py` 第 888-903 行：改密逻辑直接从接收 `new_password` 到 `generate_password_hash()` 到 `UPDATE`，中间**没有任何 `old_password` 或 `check_password_hash()` 调用**。

#### 修复方案（v6beta）

改密前必须验证当前密码，使用常量时间比较：

```python
# 必须验证原密码
if not check_password_hash(user["password"], old_password):
    return render_template(..., error="原密码错误")
```

#### 修复代码对比

| 关键点 | v6alpha | v6beta |
|--------|---------|--------|
| 旧密码校验 | 无 | `check_password_hash(user["password"], old_password)` |
| 密码强度 | 无强制 | `validate_password_strength()` |
| 确认密码 | 无 | new_password == confirm_password |

### 漏洞3：CSRF 保护缺失

#### 漏洞原理

v6alpha 的改密表单**不含 CSRF token 隐藏字段**，服务端也**不调用 `validate_csrf()`**。攻击者可构造恶意网页，当已登录用户访问时自动提交表单篡改密码。

```html
<!-- 恶意网站 evil.com -->
<form action="http://target/change-password" method="POST">
    <input name="username" value="admin">
    <input name="new_password" value="hacked123!">
</form>
<script>document.forms[0].submit();</script>
```

#### 漏洞位置

- `v6alpha/templates/profile.html` 第 54-73 行：改密表单无 `csrf_token` 隐藏字段
- `v6alpha/app.py` 第 885-903 行：POST 处理入口无 `validate_csrf()` 调用

#### 修复方案（v6beta）

表单嵌入 CSRF token 隐藏字段，服务端调用 `validate_csrf()` 验证：

```html
<input type="hidden" name="csrf_token" value="{{ csrf_token }}">
```
```python
if not validate_csrf():
    return render_template(..., error="请求校验失败，请刷新页面后重试")
```

## 二、白盒审计同步发现并修复的问题

v6beta 自审计额外确认并修复：

### 2.1 改密后旧 Session 不失效

**问题：** 改密成功后原 session cookie 仍有效（因为 session_version 未更新）。在原有代码中 `session["session_version"]` 与 DB 同步更新的逻辑存在不一致。

**修复：** 改密后不再同步更新 `session["session_version"]`——改由 `db_update_user_password()` 在 SQL 层原子递增 `session_version = session_version + 1`。`enforce_session_version` before_request 钩子在下次请求检测到版本不匹配，清除 session 并重定向到 `/login`。用户必须使用新密码重新登录。

### 2.2 改密成功后跳转优化

**问题：** 原代码改密成功后停留在成功提示页面，不直观。

**修复：** 改密成功后使用 `redirect(url_for("profile"))` 跳转到个人中心（或 `/login`，因 session 已失效）。

### 2.3 并发改密安全

经白盒审计确认：`db_update_user_password()` 使用原子 SQL `UPDATE users SET password = ?, session_version = session_version + 1`，不存在读-改-写竞态窗口。

|------|------|:--:|:--:|:--:|
| 任意用户密码篡改 | 水平越权 (IDOR) | 🔴 严重 | 可篡改任何人 | 已修复 |
| 无需验证原密码 | 认证绕过 | 🔴 严重 | 直接覆盖 | 已修复 |
