# 漏洞报告 — v6alpha 安全评估

## 一、本次重点修复 — 密码修改安全漏洞

### 漏洞1：任意用户密码篡改（无身份校验）

#### 漏洞原理

v6alpha 的 `/change-password` 路由从 POST 表单的 `username` 字段直接读取目标用户名，**未验证该用户名是否与当前登录会话（session["uid"]）匹配**。因此，任意已登录用户可提交任意 `username` 来修改他人的密码。

#### 漏洞位置

- **文件**: `v6alpha/app.py`，第 871–919 行
- **关键代码**（第 888–903 行）：

```python
# 漏洞版本：POST only，从表单获取 username，不校验原密码，不校验 CSRF
# 任何已登录用户可修改任意用户的密码
username = request.form.get("username", "")       # 第 890 行 — 身份来自表单，非 Session
new_password = request.form.get("new_password", "")

conn = get_db()
try:
    conn.execute(
        "UPDATE users SET password = ? WHERE username = ?",    # 第 902 行 — 直接以表单 username 更新
        (generate_password_hash(new_password), username)
    )
    conn.commit()
```

- **模板**: `v6alpha/templates/profile.html`，第 58–72 行

```html
<form method="POST" action="{{ url_for('change_password') }}">
    {# 故意不包含 csrf_token #}
    <input type="hidden" name="username" value="{{ user.username }}">
    ...
</form>
```

#### 攻击方式

1. Alice 登录系统（alice / Alice@2025#User!）
2. Alice 向 /change-password 发送 POST 请求：
   ```
   POST /change-password
   Body: username=admin&new_password=hacked
   ```
3. 系统执行 `UPDATE users SET password = ? WHERE username = 'admin'`，**无需验证任何权限**
4. 攻击者收到 `302 Redirect → /profile`，admin 密码已被 alice 修改

#### 修复方案（v5beta 的修复）

v5beta 从 `session["uid"]` 确定身份，并且通过 `db_get_user_full(uid)` 获取到目标用户记录，确保只能修改自己的密码。

#### 修复代码对比

| 项目 | v6alpha（漏洞版本） | v5beta（修复版本） |
|------|---------------------|-------------------|
| 身份来源 | `request.form.get("username")`（第 890 行） | `session.get("uid")`（第 873 行） |
| 更新条件 | `WHERE username = ?`（第 902 行） | `WHERE uid = ?`（第 228 行，`db_update_user_password`） |
| 用户匹配 | 不校验 session 与表单 username | 始终以 session uid 为准 |

v5beta 修复代码（`app.py` 第 905 行）：

```python
db_update_user_password(uid, generate_password_hash(new_password))
#                          ^^^ session["uid"]，不可篡改
```

#### 效果验证

```bash
# 以 alice 登录后，无需 admin 原密码即可篡改 admin 密码
curl -X POST http://127.0.0.1:8088/change-password \
  -d "username=admin&new_password=hacked"
# → 302 Redirect, admin 密码变为 "hacked"
```

### 漏洞2：无需验证原密码即可改密

#### 漏洞原理

v6alpha 的 `/change-password` 路由**完全没有验证原密码**。攻击者仅需知道目标用户名，即可直接设置新密码。

#### 漏洞位置

- **文件**: `v6alpha/app.py`，第 888–903 行
- 整个 POST 处理逻辑中没有出现过 `old_password`、`current_password` 或任何密码验证调用

v6alpha 关键代码（从第 888 行开始，无旧密码检查）：

```python
username = request.form.get("username", "")
new_password = request.form.get("new_password", "")

if not username or not new_password:
    return redirect(url_for("profile"))

conn = get_db()
try:
    conn.execute(
        "UPDATE users SET password = ? WHERE username = ?",
        (generate_password_hash(new_password), username)
    )
```

对比 v6alpha 源码，没有任何 `check_password_hash` 或 `current_password` 相关的调用。

#### 攻击方式

1. 攻击者只需知道目标用户名（如 `admin`）
2. 发送 POST 请求：
   ```
   POST /change-password
   Body: username=admin&new_password=任意密码
   ```
3. 密码即刻被修改，无需知道原密码

#### 修复方案（v5beta 的修复）

v5beta 在修改密码前要求验证原密码（除首次登录强制改密外）。通过 `check_password_hash` 比对用户当前密码哈希。

#### 修复代码对比

| 项目 | v6alpha（漏洞版本） | v5beta（修复版本） |
|------|---------------------|-------------------|
| 原密码检查 | **无** | `check_password_hash(user["password"], old_password)`（第 893 行） |
| 首次登录例外 | — | `if not is_forced:` 跳过旧密码检查（第 892 行） |

v5beta 修复代码（`app.py` 第 892–895 行）：

```python
old_password = request.form.get("old_password", "")
...
if not is_forced:
    if not check_password_hash(user["password"], old_password):
        return render_template("change_password.html",
            error="原密码错误", is_forced=False)
```

#### 效果验证

```bash
# 以 alice 登录，修改 admin 密码 — 不需要 admin 的原密码
curl -X POST http://127.0.0.1:8088/change-password \
  -d "username=admin&new_password=hacked"
# → 成功，302 Redirect
```

### 漏洞3：CSRF 保护缺失

#### 漏洞原理

v6alpha 的 `/change-password` 路由**完全没有调用 `validate_csrf()`**，且表单中**没有 `csrf_token` 隐藏字段**。攻击者可构造恶意页面，诱导已登录用户访问，该页面通过表单自动提交或 AJAX 跨站请求实现密码篡改。

#### 漏洞位置

- **文件**: `v6alpha/app.py`，第 871–919 行
  - 整个 change_password 函数中没有任何 `validate_csrf()` 调用
- **文件**: `v6alpha/templates/profile.html`，第 58–72 行
  - 表单中没有 `<input type="hidden" name="csrf_token">`
- **文件**: `v6alpha/templates/change_password.html`，第 25 行
  - 强制改密表单中也已移除 CSRF token

v6alpha 路由（第 871–919 行，无 CSRF 校验）：

```python
@app.route("/change-password", methods=["GET", "POST"])
def change_password():
    uid = session.get("uid")
    if not uid:
        return redirect(url_for("login"))
    ...
    username = request.form.get("username", "")
    new_password = request.form.get("new_password", "")
    ...
    # 此处未调用 validate_csrf() — 无 CSRF 保护
```

v5beta 的修复对比（第 884 行）：

```python
if request.method == "POST":
    if not validate_csrf():    # ← v5beta 有 CSRF 校验
        return render_template("change_password.html",
            error="请求校验失败，请刷新页面后重试", is_forced=is_forced)
```

#### 攻击方式

攻击者构造以下恶意 HTML 页面，诱导已登录用户访问：

```html
<!DOCTYPE html>
<html>
<body>
  <h1>点击领取优惠券</h1>
  <form id="csrf-form" method="POST" action="http://目标系统:8088/change-password">
    <input type="hidden" name="username" value="admin">
    <input type="hidden" name="new_password" value="attacker_pwd">
  </form>
  <script>document.getElementById('csrf-form').submit();</script>
</body>
</html>
```

当管理员 admin 已登录目标系统且未退出时，访问上述页面会自动提交表单，将 admin 的密码修改为 `attacker_pwd`。

#### 修复方案（v5beta 的修复）

v5beta 使用了双保险：
1. 服务端调用 `validate_csrf()` 校验请求中的 token 与 session 中存储的 token 是否一致
2. 表单中通过 `{{ csrf_token }}` 渲染隐藏字段，token 由 `generate_csrf_token()` 生成

#### 修复代码对比

| 项目 | v6alpha（漏洞版本） | v5beta（修复版本） |
|------|---------------------|-------------------|
| 服务端校验 | **无** `validate_csrf()` 调用 | 第 884 行：`validate_csrf()` |
| 表单 CSRF 字段 | **无** | `<input type="hidden" name="csrf_token" value="{{ csrf_token }}">` |
| Token 生成 | **无** | `generate_csrf_token()`（第 913 行） |

#### 效果验证

```bash
# POST 请求中不携带 csrf_token — 仍成功修改密码
curl -X POST http://127.0.0.1:8088/change-password \
  -b "session=..." \
  -d "username=admin&new_password=PwnedNoCSRF"
# → 302 Redirect, 成功！CSRF 保护完全缺失
```

## 二、其他同步发现并修复的问题

v6alpha 基于 v5beta 代码库构建，除上述 `/change-password` 相关的三个漏洞外，**其余安全机制与 v5beta 保持一致且未被弱化**，具体包括：

| 安全机制 | v5beta | v6alpha | 状态 |
|----------|--------|---------|------|
| 参数化 SQL 查询 | 全部使用 `?` 占位符 | 全部使用 `?` 占位符 | 一致（安全） |
| 密码哈希存储 | `generate_password_hash` | `generate_password_hash` | 一致（安全） |
| 登录限流（IP+用户名双维度） | 第 446–591 行 | 第 446–591 行 | 一致（安全） |
| 时序侧信道防御（假哈希） | 第 457 行 | 第 457 行 | 一致（安全） |
| Session HTTPS-only 配置 | 第 354–360 行 | 第 354–360 行 | 一致（安全） |
| Session 版本管理（改密后旧 session 失效） | 第 659–669 行 | 第 659–669 行 | 一致（安全） |
| CSRF 保护（登录/注册/充值/上传/个人信息修改） | 全部有 validate_csrf() | 全部有 validate_csrf() | 一致（安全） |
| XSS 防御 — 模板上下文注入安全 | 使用 Jinja2 自动转义 | 使用 Jinja2 自动转义 | 一致（安全） |
| XSS 防御 — 输入过滤 | 第 614–626 行 | 第 614–626 行 | 一致（安全） |
| 文件上传 — 扩展名白名单 | 第 369 行 | 第 369 行 | 一致（安全） |
| 文件上传 — MIME 类型校验 | 第 371 行 | 第 371 行 | 一致（安全） |
| 文件上传 — 魔数验证 | 第 379–407 行 | 第 379–407 行 | 一致（安全） |
| 文件上传 — UUID 文件名防枚举 | 第 1139 行 | 第 1139 行 | 一致（安全） |
| 路径穿越防护 — `/page` 路由 | 第 1186–1211 行 | 第 1186–1211 行 | 一致（安全） |

| Debug 模式安全警告与限制 | 第 316–335 行 | 第 316–335 行 | 一致（安全） |

### 总结

v6alpha **仅有的三个漏洞**均集中在 `/change-password` 路由及其对应的前端表单上，属于**故意引入的安全缺陷**。其他安全机制（CSRF 保护在其他路由上仍然有效、参数化查询、密码哈希存储、登录限流、文件上传校验、路径穿越防护等）均未受影响。

修复这三个漏洞后（即回退到 v5beta 的实现），系统整体安全性恢复到 v5beta 的基准水平。
